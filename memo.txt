# Meccha Camouflage memo

This memo records practical findings from the camouflage experiments. It is not a design document. It exists to prevent repeating failed debugging loops.

## 1. Current main state

- Runtime architecture: C++ controller EXE + injected bridge DLL.
- Controller source: `runtime/src/meccha_runtime_controller.cpp`.
- Bridge source: `runtime/src/meccha_xenos_bridge.cpp`.
- Minimal runtime SDK: `runtime/sdk/meccha_sdk_min.hpp`.

## 2. Main conclusion

The current implementation does not have exact mesh render data or true UV chart data.

Missing data:

- `FSkeletalMeshRenderData`.
- `FSkeletalMeshLODRenderData`.
- Vertex buffer.
- Index buffer.
- UV buffer.
- Section/material index.
- Skinned current pose.
- UV island adjacency / chart map.

Because this data is missing, the runtime cannot reliably answer:

- Which triangle owns a painted UV.
- Which UV island a texel belongs to.
- Whether two adjacent texels in atlas space belong to the same body part.
- Whether a fill operation crosses from arm to torso, torso to leg, or left to right limb.

Practical effect:

- Holes decrease when fill is aggressive.
- Cross-region color contamination increases when fill is aggressive.
- Contamination decreases when fill is strict.
- Holes increase when fill is strict.

This is not primarily a color-space problem. It is a topology problem.

## 3. Side surface area and grazing-angle problem

Side quality is not only a UV chart issue.

The side of the body has larger real surface area than it appears to have from the current camera. Some side surfaces are almost parallel to the camera ray. In that case, screen-space sampling sees them as very thin strips even though their actual mesh/UV area is large.

Approximate projection relationship:

```text
screen_area ~= surface_area * max(0, dot(normal, view_dir)) / depth^2
```

When `dot(normal, view_dir)` is near zero:

- The surface is grazing/tangent to the view.
- It has very little screen area.
- A screen-space sampler will produce too few hits.
- Paint centers become unstable or unavailable.
- Texture/paint coverage becomes dotted or sparse.

Consequence:

- Increasing front sample count does not fix side coverage.
- Increasing splat radius hides holes but creates visible palette-like dots.
- Side sampling must be based on mesh surface area / UV area, not only screen-space area.
- This again points to the need for exact mesh render data and UV chart information.

- External repo offsets are useful for context validation.
- They are not enough to implement correct full-body camouflage.

## 7. Route findings

### 7.1 Local texture import diagnostics

Purpose:

- Quickly display generated atlas bytes in-game.
- Avoid slow stroke streaming.

What worked:

- Fast local visual feedback.
- Confirmed `ImportChannelFromBytes` can mutate local channels.

What failed:

- Local import is not multiplayer proof.
- It created confusion because local visual success looked like route success.
- It did not solve side/back source correctness.

Decision:

- Do not treat local-only import as success.
- If used, it must be explicit diagnostic-only.

### 7.2 Texture sync strict probe

Purpose:

- Use texture import plus texture sync RPC as a fast multiplayer candidate.

What worked:

- Much faster than dense `ServerPaintBatch` streaming.
- Local import and sync RPC dispatch completed.
- Good enough to remain a candidate if remote sync is verified.

What remains unverified:

- Whether a remote client sees the texture sync result.
- One local client cannot prove multiplayer replication.

Known problems:

- If atlas/source data is wrong, texture sync applies the wrong result quickly.
- Side/back quality remains bad without true mesh/UV source.

### 7.3 ServerPaintBatch / replicated paint stream

Purpose:

- Apply paint through replicated paint API.

What worked:

- `ServerPaintBatch` dispatch can send many strokes.
- Some visual paint results were observed.

What failed:

- Dense strokes are slow.
- Stroke count reduction hurts quality.
- Some runs showed `server_sent > 0` while exported albedo hash did not change.
- Visual layer and `ExportChannelToBytes` observation may not always match.

Important:

- Paint stream cannot fix wrong source placement.
- If atlas or UV placement is wrong, paint reproduces the same wrong content more slowly.

### 7.4 Front-only route

What worked:

- Front sampling became much faster after cached-triangle hit-test use.
- Front-only is more stable than side/back.

What failed:

- It is not whole-body camouflage.
- Curved edges and side transition remain weak.

### 7.5 Full side/back virtual-view route

Purpose:

- Generate side/back samples using virtual camera/view sampling.

What worked:

- Some side/back samples can be produced.
- Increasing attempts can increase coverage.

What failed:

- Back often remains sparse.
- Side/back can become dotted.
- Large splat radius creates visible palette-like spots.
- Fill can move colors between unrelated body parts.
- More attempts/views increased runtime and did not reliably solve correctness.

Reason:

- Screen-space sampling is a poor proxy for UV/surface-area coverage at grazing angles.
- No true chart boundary exists to constrain fill.

### 7.6 Static multi-view edge-weighted hybrid

Intended formula, conceptually:

```text
A_dom   = dominant front high-frequency color
A_multi = low-frequency color from multiple views
E       = edge/silhouette/grazing weight
A_final = low_frequency(A_multi) + alpha(E) * high_frequency(A_dom)
```

Purpose:

- Keep front detail.
- Stabilize side/edge with lower-frequency multi-view color.
- Avoid fully solving rear/back initially.

What worked:

- Some holes became filled.
- Some side areas looked more continuous than earlier dotted output.

What failed:

- Cross-region color contamination remained.
- Boundary areas still got unexpected colors.
- Side/back could still be dotted, sparse, or wrong.
- Holes vs contamination became a tradeoff.

Reason:

- The method needs correct chart/island boundaries.
- The implementation currently uses observed/proxy data, not true mesh topology.

Conclusion:

- The idea is not necessarily invalid.
- The current implementation is not sufficient.
- More weight tuning without true mesh data is low-value.

## 8. Color and material findings

### 8.1 sRGB / linear

Standard transform:

```text
C_srgb = byte / 255
if C_srgb <= 0.04045:
    C_linear = C_srgb / 12.92
else:
    C_linear = ((C_srgb + 0.055) / 1.055) ^ 2.4
```

Findings:

- sRGB/linear changes can make output look blue-ish, dark, red-ish, or washed out.
- It does not explain arm/leg/torso cross-region contamination.
- Color transfer is a later calibration problem, not the main topology problem.

### 8.2 Metallic / roughness

Experiments included:

- Full-body white metallic base.
- Metallic 1.0 / roughness 0.0.
- Metallic 0.0.
- High roughness around 0.92.
- Importing or preserving material channels.

Findings:

- Material values change perceived correctness.
- Metallic can hide or exaggerate albedo errors.
- Roughness can reduce reflective confusion.
- Material settings do not fix wrong UV/source placement.

Current preference for diagnostics:

```text
metallic = 0.0
roughness ~= 0.92
```

## 9. Capture / bulk readback findings

Original problem:

- Per-pixel readback was too slow.
- Bulk readback initially failed with zero candidates.

Fix that worked:

- Do not assume `FArrayProperty.ElementSize == sizeof(TArray)`.
- Read the `TArray` header at the parameter property offset.
- Decode `ReadRenderTargetRaw` as `TArray<FLinearColor>` candidate.
- Decode `ReadRenderTarget` as `TArray<FColor>` candidate.
- Keep `TArray<uint8>` only as auxiliary candidate.

Calibration idea:

```text
candidate image = bulk readback candidate
known samples = small set of screen/capture sample colors
error = median RGB error over matched points
choose candidate/flip/color-transform with lowest median error
```

What this solved:

- Front capture source became faster and more stable.
- The route moved closer to the older high-quality texture path.

What it did not solve:

- Side/back surface area and UV island contamination.

## 10. Crash and failure notes

### 10.1 Game update / SDK mismatch crash

Observed symptoms:

```text
[WinError 10054] connection forcibly closed
empty_bridge_response
EXCEPTION_ACCESS_VIOLATION reading address 0x0
```

Likely meaning:

- Injected bridge touched stale offsets or invalid UE object layout.
- Game update changed SDK/layout.

Action:

```text
make sdk-dump
```

Then review:

- `dumper-sdk/`
- `runtime/sdk/meccha_sdk_min.hpp`
- `runtime/sdk/meccha_mesh_layout.hpp`

### 10.2 `world_unavailable` / `local_pawn_unavailable`

Meaning:

- Game is in menu/loading state.
- Pawn or world context is not ready.
- Not necessarily a crash.

### 10.3 `empty_bridge_response`

Meaning:

- Controller connected but bridge returned no JSON.
- Usually suggests bridge/game crash during processing.

### 10.4 SceneCapture / D3D instability

Known note:

- Earlier D3D/backbuffer/SceneCapture experiments caused instability.
- Failure labels such as `front_capture_backend_disabled_after_d3d12_crash` remain in code/history.

Practical rule:

- Treat SceneCapture and render-target offsets as crash-sensitive after game updates.

### 10.5 `texture_sync_api_unavailable`

Meaning:

- No texture sync RPC candidate was found.
- Strict texture sync route should stop before local import in this case.

Reason:

- Local-only import would look successful but would not be a multiplayer route.

## 12. What not to repeat

Do not repeat these as primary fixes:

- Increase side/back view count only.
- Increase sample count only.
- Increase splat radius only.
- Fill across unknown UV boundaries.
- Mirror front to back as a default strategy.
- Tune sRGB/linear to fix cross-part contamination.
- Tune metallic/roughness to hide atlas errors.
- Convert wrong atlas to paint stream and expect correctness.
- Memory scan arbitrary candidate buffers and treat them as topology without deterministic validation.

## 13. Best next direction

The next serious technical direction is exact mesh render-data access.

Required path:

1. Read deterministic mesh render data.
2. Extract vertex/index/UV/section data.
3. Build CPU-side surface and UV chart map.
4. Validate CPU raycast UV against a few `HitTestAtScreenPosition` oracle samples.
5. Sample by mesh surface area / UV area, not only screen-space area.
6. Fill only inside true UV islands.
7. Apply with texture sync if remote sync is verified, otherwise fall back to paint stream only after atlas correctness is proven.

Short judgment:

- Front-only: usable baseline, incomplete.
- Texture sync: fastest candidate, remote multi unverified.
- ServerPaintBatch: multiplayer-oriented but slow for dense output.
- Static hybrid: concept may be valid, current implementation lacks true topology.
- Exact mesh/UV/chart decode: main blocker for side/back quality.
